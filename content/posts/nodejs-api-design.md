设计一个好用的 API，是后端开发中非常核心的能力。对前端工程师来说，理解 API 设计也能帮助我们更好地和服务端协作。

## RESTful 设计原则

RESTful API 的核心是围绕资源建模，并合理使用 HTTP 方法。

- `GET`：读取资源
- `POST`：创建资源
- `PUT` / `PATCH`：更新资源
- `DELETE`：删除资源

URL 应该表达资源，而不是动作。例如 `/api/posts/123` 通常比 `/api/getPostById?id=123` 更清晰。

## 状态码

状态码要准确表达请求结果。

- `200`：请求成功
- `201`：资源创建成功
- `400`：请求参数错误
- `401`：未认证
- `403`：无权限
- `404`：资源不存在
- `500`：服务端错误

## 错误响应

错误响应建议保持统一格式，方便前端处理。

```json
{
  "message": "参数无效",
  "code": "INVALID_PARAMS",
  "details": {}
}
```

## 项目结构

一个清晰的 Node.js API 项目通常会把路由、控制器、服务和数据访问层分开。这样可以让业务逻辑更容易测试和复用。

## 总结

好的 API 应该稳定、清晰、可预测。它既是服务端能力的边界，也是前后端协作的契约。
